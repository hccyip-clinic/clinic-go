# Phase 1 Vertical Slice Wayfinder

## Destination

Reach a runnable Phase 1 vertical slice: a localhost Go server with SQLite migration, dashboard, and end-to-end patient and receipt CRUD. HTMX polish, reports, backups, and other Phase 2+ work follow only after this slice is coherent.

## Notes

The route follows `docs/QUICKSTART.md`, `docs/prototype-spec.md`, and `CONTEXT.md`. Use domain vocabulary from `CONTEXT.md`; preserve server-rendered Go templates, SQLite, and focused incremental changes.

## Decisions so far

- The destination is the Phase 1 vertical slice, not a dashboard-only implementation.
- Draft receipts do not receive a receipt number; numbering occurs on finalization.
- The existing `settings` table remains canonical for Phase 1, with configuration aligned to it.
- Normalized HKIDs remain available at rest for lookup, with hashes for deduplication and masking for exports/backups.
- [Choose and migrate to a pure-Go SQLite driver](001-sqlite-driver-migration.md) — use `modernc.org/sqlite` v1.54.0 with the existing database wrapper and pragmas.
- [Define the Phase 1 receipt lifecycle](002-receipt-lifecycle.md) — persist editable drafts, finalize atomically with unique numbering, then make finalized receipts immutable.
- [Define patient identity and validation](003-patient-identity-and-validation.md) — normalize and validate canonical HKIDs, enforce uniqueness, preserve Unicode names, and keep HKIDs immutable.
- [Shape the first server-rendered UI slice](004-first-slice-ui-boundary.md) — use the guided flow as the first production UI boundary, with drafts available before finalization.
- [Vendor HTMX for offline deployment](005-vendor-htmx.md) — use npm package v2.0.7 and serve the minified asset locally.

## Not yet specified

- How existing installations migrate from the original receipt schema to nullable draft receipt numbers.
- Whether the first receipt UI supports multiple line items or deliberately limits the slice to one.
- Which authentication, session, and CSRF guarantees are required before Phase 1 is considered shippable.
- Whether Tailwind CSS is part of the Phase 1 runtime artifact or deferred until the UI is stabilized.
- How finalized receipts enter the archived state and how retention enforcement is triggered.
- Which quickstart/prototype documentation needs reconciliation after implementation decisions land.

## Out of scope

- Reports, exports, auto-backup, retention jobs, notifications, and dashboard metrics beyond what is needed to prove the slice.
- Multi-user roles and payment, insurance, or appointment integrations.
